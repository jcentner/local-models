# 2026-06-14 — Kickoff: turning a tweet into a workshop

The starting point was a single file: a copy of Andrej Karpathy's post about
using LLMs to maintain personal knowledge bases. The goal for this session was to
turn that idea — plus a laptop and a vague ambition to "learn how local models
actually work" — into something organized enough to compound over time.

## What I set out to do

Build a one-stop place to run models locally, study how they work (quantization,
optimization, serving stacks), run my own benchmarks, and keep a real record of
what I do and learn — narrative included, because this should feed blog and
Twitter posts, not just rot in a terminal scrollback. GitHub Copilot (VS Code /
CLI) is the agent driving the research; the local models are the subject, not the
assistant.

## What I learned today

**The framework has a canonical source.** The original file was just Karpathy's
tweet, but he followed it with a gist, `llm-wiki.md`, that pins down the actual
pattern: three layers — immutable `raw/` sources, an LLM-compiled `wiki/`, and a
schema file (`AGENTS.md` / `CLAUDE.md`) — driven by an **ingest → query → lint**
loop, with a content `index.md` and an append-only, greppable `log.md`. The key
reframe vs RAG: compile knowledge once and keep it current, so it compounds,
instead of re-deriving it on every question. At this scale (hundreds of pages,
well under ~100k tokens) plain markdown beats a vector DB on both reliability and
simplicity. I also folded in a project-oriented half — a `lab/` — for my own runs
and decisions, which is the "memwiki" flavor several people in the gist comments
described.

**The machine is more constrained than the spec sheet suggests, in two specific
ways.** The GPU is the obvious limiter — RTX 5070 Laptop, 8 GB — which puts ~7–9B
models at 4-bit fully on the card and pushes anything bigger into slow CPU/GPU
hybrid. But the surprise was RAM: this is **WSL2 on Windows 11**, and WSL caps the
guest at ~50% of host memory by default, so Ubuntu sees only **~15 GB of the 32 GB**
installed. There was no `.wslconfig` at all. That matters immediately, because the
first model I want to try needs ~18 GB.

**DiffusionGemma is a real "first experiment," not a warm-up.** It's a diffusion /
block-autoregressive model — it denoises a canvas of tokens in parallel rather
than going left-to-right — so it won't run on stock Ollama at all; it needs
Unsloth Studio or a special llama.cpp branch. And at ~18 GB it doesn't fit until I
raise the WSL memory cap. So before I can even start, two prerequisites: bump
`.wslconfig`, and pick a diffusion-capable runner. Exactly the kind of friction
worth documenting.

**Blackwell's quiet tax: CUDA ≥ 12.8.** The 8 GB card is `sm_120`; its kernels
only exist in recent toolkits and wheels. Ollama dodges this by bundling its own
runtime (and indeed it already works), but anything I build from source —
llama.cpp's diffusion branch, vLLM, torch — will need a 12.8+ toolkit, and there's
no CUDA toolkit installed yet. Good to know before I hit a cryptic "no kernel
image" error.

## Decisions

- **Organization:** `raw/` → `wiki/` → schema, plus `lab/{journal,experiments,benchmarks}`,
  `env/`, `scripts/`. Obsidian-friendly markdown so I can browse the graph later.
- **Daily driver:** Ollama (already installed, 0.20.2, with two qwen3.5 models).
  llama.cpp for engine-level control and benchmarks; vLLM is a stretch/post-upgrade
  goal; Lemonade + the NPU are a separate Windows-side experiment (WSL can't see
  the NPU).
- **Repo visibility:** start **private**, flip public later. The asymmetry decides
  it — private→public is a trivial one-way door that keeps history; public→private
  after leaking a token or committing a copyrighted clipped article is too late.
  `raw/` is git-ignored and secrets are gitignored so the repo stays public-ready.
- **Security baked in:** `raw/` is an indirect-prompt-injection surface, so source
  content is treated as data, never instructions.
